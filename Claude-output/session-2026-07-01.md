## Mental Model for Revision

### 1. GitOps — the three principles

**The problem with `kubectl apply`:**
The cluster becomes the source of truth. Someone runs `kubectl edit deployment` directly — your Git repo no longer reflects reality. No audit trail, no rollback, no drift detection.

**GitOps flips this — Git is the single source of truth:**

| Principle | What it means |
|---|---|
| Declarative | Everything the cluster runs is described in Git as YAML |
| Versioned | Every change is a Git commit — full history, diff, rollback |
| Reconciliation loop | ArgoCD continuously watches Git + cluster; if they drift, it corrects the cluster back |

ArgoCD is like a Kubernetes controller — the same pattern as a Deployment controller watching for pod drift. ArgoCD watches for *manifest drift* and corrects it.

---

### 2. CI vs CD split in GitOps

**Critical distinction — CI pipeline no longer runs `kubectl apply`:**

```
Developer pushes code
    → CI Pipeline (GitHub Actions / Azure Pipelines):
        build → test → push image to registry → UPDATE IMAGE TAG IN GIT
                                                          ↓
                                               ArgoCD detects Git change
                                                          ↓
                                               ArgoCD syncs cluster (kubectl apply)
```

- **CI** = build, test, push image, write to Git. Never touches the cluster directly.
- **CD** = ArgoCD watches Git, applies to cluster. Only thing that touches `kubectl`.

The pipeline's only connection to the cluster is **through Git**. This makes every deployment auditable — no "someone ran kubectl from their laptop at 2am" without a Git trace.

---

### 3. ArgoCD components — what each does

| Pod | Kind | Role |
|---|---|---|
| `argocd-application-controller-0` | StatefulSet | Core reconciliation loop — watches cluster state, detects drift |
| `argocd-applicationset-controller` | Deployment | Generates ArgoCD Applications from templates (multi-env) |
| `argocd-dex-server` | Deployment | SSO/OIDC authentication (Dex identity provider) |
| `argocd-notifications-controller` | Deployment | Sends Slack/email/webhook notifications on sync events |
| `argocd-redis` | Deployment | Cache layer — stores application state; re-derived from Git+cluster if Redis restarts |
| `argocd-repo-server` | Deployment | Clones Git repos, renders Helm/Kustomize/plain YAML |
| `argocd-server` | Deployment | API server + web UI |

**Why application-controller is a StatefulSet (interview answer):**
Stable pod identity (`-0`, `-1`, `-2`) is needed for:
1. **Leader election** — in HA mode, multiple replicas run but only one leads; `-0` is typically leader
2. **Cluster sharding** — when managing hundreds of clusters, each StatefulSet pod owns a slice (pod-0 → clusters 1–50, pod-1 → clusters 51–100)

**NOT** for persistent storage — Redis handles caching, Git + K8s API are the sources of truth.

---

### 4. OutOfSync vs Healthy — what the combination means

| Sync Status | Health | Meaning |
|---|---|---|
| OutOfSync | (blank) | Resource does NOT exist in cluster yet |
| OutOfSync | Healthy | Resource EXISTS in cluster but config differs from Git — this is drift |
| Synced | Healthy | Resource exists and matches Git exactly ✅ |
| Synced | Progressing | Resource created/updated, pods still starting up |

---

### 5. ArgoCD sync — what happens under the hood

`argocd app sync` sends `kubectl apply` for each resource — patches only fields that differ, doesn't delete/recreate. Pods don't restart unless the pod spec changed.

ArgoCD pins each sync to a **Git commit hash** (`Synced to 0b3b3ae`). Every cluster state change is traceable to an exact commit — full audit trail.

**Sync policy:**
- `Manual` (Day 1) — ArgoCD detects drift but waits for you to approve sync
- `Automated` (Day 2) — ArgoCD syncs automatically when Git changes

### 6. Key CLI commands

```bash
# Install ArgoCD on cluster
kubectl create namespace argocd
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/v3.4.4/manifests/install.yaml

# Get initial admin password
argocd admin initial-password -n argocd

# Port-forward UI
kubectl port-forward svc/argocd-server -n argocd 8080:443
# Open https://localhost:8080

# Login via CLI
argocd login localhost:8080 --username admin --password <pwd> --insecure

# Register a Git repo
argocd repo add https://github.com/rmonde/DevOpsWithAI.git

# Create an Application
argocd app create devopswithai \
  --repo https://github.com/rmonde/DevOpsWithAI.git \
  --path k8s \
  --dest-server https://kubernetes.default.svc \
  --dest-namespace default \
  --sync-policy manual

# Check app status
argocd app get devopswithai

# Sync manually
argocd app sync devopswithai
```

## Interview Questions This Session Covers

**"What is GitOps and how is it different from traditional CI/CD?"**
→ GitOps makes Git the single source of truth for cluster state. Traditional CI/CD pipelines run `kubectl apply` directly. In GitOps, the pipeline only writes to Git — ArgoCD (the CD tool) watches Git and applies changes to the cluster. Every cluster change has a Git commit. Rollback = `git revert`.

**"What does ArgoCD do when it detects drift?"**
→ With manual sync policy it alerts you — you see `OutOfSync` in the UI/CLI. With auto-sync + self-heal enabled, it immediately runs `kubectl apply` to restore the desired state from Git. Someone running `kubectl edit` directly gets overwritten within seconds.

**"Why is the ArgoCD application controller a StatefulSet?"**
→ Stable pod identity for leader election (only one controller leads in HA) and cluster sharding (each pod owns a subset of managed clusters). Unlike Prometheus, it doesn't need PVCs — Redis handles caching.

**"What's the difference between Sync Status and Health Status in ArgoCD?"**
→ Sync Status compares Git desired state vs cluster actual state (Synced/OutOfSync). Health Status reflects whether the resource is functioning correctly (Healthy/Progressing/Degraded). A resource can be Synced but Degraded (applied correctly but app is crashing). A resource can be OutOfSync but Healthy (drifted but still running fine).
